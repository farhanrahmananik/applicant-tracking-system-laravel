# ATS Schema Blueprint

## 1. Blueprint Conventions

This blueprint targets Laravel 12, MySQL 8+, InnoDB, `utf8mb4`, and strict mode. It is a proposal for future migrations, not executable migration code.

- `id`: `BIGINT UNSIGNED AUTO_INCREMENT` primary key, equivalent to Laravel `id()`.
- Foreign keys: `BIGINT UNSIGNED`, equivalent to `foreignId()`.
- `public_id`: ULID stored as `CHAR(26)` and generated by the application before insert.
- `created_at`, `updated_at`: nullable Laravel timestamps unless explicitly stated otherwise.
- `deleted_at`: nullable timestamp where soft deletion is proposed.
- Workflow values: `VARCHAR`, never MySQL `ENUM`.
- Date-time values: UTC. Schedule rows also retain an IANA time zone.
- Money: `DECIMAL(15,2)` plus `CHAR(3)` currency.
- Actor references: nullable and `ON DELETE SET NULL`.
- Tenant/business aggregate references: normally `ON DELETE RESTRICT`.
- True dependent rows and pivots: normally `ON DELETE CASCADE`.

Index names below are descriptive suggestions. Laravel-generated names are acceptable when they remain unambiguous and within MySQL limits.

## 9. Cross-Table Invariants

Ordinary foreign keys cannot enforce all tenant and workflow rules. Future Services must enforce these rules inside transactions, with row locks where concurrent writes could conflict:

- Department, job post, candidate, application, interview, feedback, offer, and notification references must resolve within the same company.
- Department managers, recruiters, organizers, and interviewers must hold appropriate company access.
- An application resume must belong to the application's candidate.
- Interview feedback reviewers must exist in `interview_schedule_user`.
- Only one candidate resume is primary at a time.
- Only one accepted offer exists per application.
- Status changes follow defined transitions and write history/audit records atomically.
- Soft-deleted parents cannot receive new active children.

Future validation belongs in Laravel Form Requests, workflow orchestration belongs in Services, and authorization belongs in Policies/Gates. Controllers should only coordinate those layers and return responses.
